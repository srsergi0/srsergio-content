# The Best Time to Build a Thing Is When It Doesn't Exist

I was standing in line at a print shop, holding a PDF of a zine I wanted to turn into a booklet. The guy behind the counter told me it would cost $40 and take three days.

I went home, wrote a Python script in an afternoon, and had my booklet printed that night.

Later, I needed to extract my flashcards from Anki. I searched for a tool. Nothing clean existed. So I built one.

I wanted to play UNO on my smartwatch. No games existed for it. So I built one.

I wanted an AI assistant to control my music while I coded. No integration existed. So I built one.

The pattern is always the same: I encounter a void, I look for a solution, and when I don't find one, I build it.

## The Void as Opportunity

Most people see a missing tool and think "someone else will build it." I see a missing tool and think "nobody else needs this enough to build it, which means I'm the only person who cares, which means I should build it."

This is not arrogance. It's arithmetic. If a problem is common, someone has already solved it. If a problem is uncommon, the solution probably doesn't exist yet. And if the solution doesn't exist, the only way to get it is to build it yourself.

The void is not a limitation. It's an invitation.

## Why Small Problems Are Worth Solving

There's a myth that you should only build things that scale. That a tool used by ten people is not worth building. That a problem faced by a niche is not a real problem.

I reject this entirely. The best projects I've built are the ones that solve problems for exactly one person: me.

And here's the thing: when you solve your own problem, you solve it well. You know the edge cases because you've hit them. You know the workflow because you live it. You know what matters and what doesn't because you've felt the pain.

A tool built for ten users, used daily by those ten users, is more valuable than a tool built for ten thousand users, used once a month by most of them.

## The Fear of Building

People don't build things because they're afraid. Afraid it's not original. Afraid it's not good enough. Afraid nobody will use it. Afraid they'll waste their time.

These fears are based on a false premise: that the purpose of building is to be validated by others.

The purpose of building is to solve a problem. If the problem is yours, the validation is the solution. You don't need users. You don't need stars. You don't need a Product Hunt launch. You need the thing to work.

Once it works, you can decide whether to share it. But the decision to build should never depend on the decision to share.

## The Compound Effect

Building small things compounds. The PDF tool taught me about document processing. The flashcard extractor taught me about SQLite. The watch game taught me about wearable development. The MCP server taught me about protocol design.

None of these were planned learning experiences. They were side effects of solving problems. But the knowledge accumulated. Each project made the next one faster, better, more ambitious.

You don't get good by studying. You get good by building things that don't exist and figuring out why they break.

## The Portfolio of Problems

Over time, you build a portfolio of solved problems. Some are small scripts. Some are full platforms. Some are used by thousands. Some are used by one. But they're all yours.

This portfolio is more valuable than any resume. It shows not what you know, but what you've done. It demonstrates not your skills, but your judgment. It proves not your competence, but your initiative.

When someone asks "can you build this?" you don't answer with your credentials. You answer with your GitHub profile. "I built something similar last year. Here's how it works. Here's what I learned. Here's how I'd adapt it."

## The Permission Problem

There's a subtle cultural barrier to building things: the belief that you need permission. Permission from an employer. Permission from a market. Permission from an audience. Permission from yourself.

You don't need permission. You need a problem and the willingness to solve it.

The most innovative software in the world was built by people who didn't ask if they should. They saw a void and filled it. Linux was built because someone wanted a free OS. Git was built because someone wanted better version control. React was built because someone wanted a better way to build UIs.

None of these started as products. They started as solutions to personal problems.

## How to Start

The next time you encounter a problem, ask three questions:

1. **Does a solution exist?** Search. Seriously. Look for existing tools. If something exists, use it.

2. **If not, can I build it?** Estimate the scope. A few hours? A weekend? A month? Be honest.

3. **Is the problem worth solving?** Not "will this make money?" but "will this make my life better?" If yes, build it.

Then build it. Don't plan it to death. Don't design the perfect architecture. Don't worry about scaling. Build the smallest version that solves your problem. Use it. Iterate. Improve.

If it's useful to you, it might be useful to others. Share it. If it's not, you've still learned something and solved your problem. Either way, you win.

## The Best Time

The best time to build a thing is when it doesn't exist. Not because you'll get rich. Not because you'll get famous. But because the void is the only place where you can build something truly yours.

Everything else is just optimization.
